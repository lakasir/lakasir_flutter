## Implementation

### Fix 1: Update success product redirect to list
- Changed `edit()` method in `product_add_controller.dart` to use `Get.until((route) => Get.currentRoute == '/menu/product')` instead of `Get.back(); Get.back();` to reliably navigate back to the product list from the edit screen (Edit → Detail → List).

### Fix 2: Toast success create/update product now appears
- In the `edit()` method, added error snackbar display (`Get.rawSnackbar`) in the catch block for `ValidationException` errors. Previously, the edit catch block didn't show any error toast.
- The toast for successful operations was already present in both `create()` and `edit()` methods but the `edit()` navigation issue (fix 1) could have caused the toast to appear on the wrong screen.

### Fix 3: Validation improvements
- Added `validateName()` method with required field validation
- Added `validateSellingPrice()` method that validates: (a) required field, (b) selling_price >= initial_price
- Added `validateInitialPrice()` method with required field validation
- Added `validateUnit()` method with required field validation
- Updated `ProductForm` to use custom validators on: product name, initial price, selling price, and unit fields
- Added `mandatory: true` flag to initial price and selling price fields
- Added translation keys `validation_selling_price_greater_than_initial_price` for both English and Indonesian

### Fix 4: Auto generation of barcode and SKU
- Added `_generateSku()` and `_generateBarcode()` methods to `ProductAddEditController` implementing the PHP logic:
  - Takes first 3 characters of category name (uppercase) as prefix
  - Takes first 3 characters of product name (uppercase) as second segment
  - Takes absolute value of product ID, padded to 4 digits with leading zeros
  - Format: `{PREFIX}-{NAME}-{ID}`
- Added `_getCategoryName()` helper method to look up category name from `CategoryController`
- In `create()` method: auto-generates SKU and barcode if fields are empty after product creation, then updates the product
- In `edit()` method: auto-generates SKU and barcode if fields are empty after product update, then updates the product

### Fix 5: Image upload - save to local storage with compression
- Modified `MyImagePicker` widget (`lib/widgets/image_picker.dart`):
  - Removed server upload functionality (`_uploadImage` method and `usingLocalImage` conditional)
  - Added `_saveAndCompressImageLocally()` method that:
    - Reads the cropped image file
    - Decodes it using the `image` package
    - Resizes images wider than 512px
    - Encodes as JPEG at quality 70 for compression
    - Saves to app documents directory with unique filename (`img_{timestamp}.jpg`)
  - Now passes local file path instead of server URL to `onImageSelected` callback
  - Updated `_buildImage()` helper method to display both local file paths and network URLs
- Updated `ProductAddEditController` to handle image saving:
  - Added `_saveAndCompressImage()` method for compressing and saving images
  - In both `create()` and `edit()` methods, checks if photoUrl is a local file, and if so, compresses and saves it
  - Skips re-compression for images already saved to app directory (contains 'product_' in path)
- Updated `BuildListImage` widget (`lib/widgets/build_list_image.dart`) to handle local file paths in addition to network URLs
- Updated `DetailScreen` (`lib/screens/products/detail_screen.dart`) to display both local and network images using `_buildProductImage()` helper method
- Set `usingLocalImage: true` on `MyImagePicker` in `ProductForm` to enable local saving mode

### Fix 6: Create product still not redirect/toast after first validation error
**Root cause**: After the first submit hits a validation error (client-side or server-side), the second successful submit would fail to redirect and show toast due to multiple issues:

1. **`catch` block silently swallowed non-ValidationException errors**: The `catch (e)` block only handled `ValidationException`. If any step after `validate()` threw a different exception (e.g., from image compression, DB operations, `int.parse`, etc.), the code entered the catch block, called `isLoading(false)`, but showed NO feedback to the user — no error toast, no redirect. The user would see the button return to normal but nothing else happened.

2. **No double-submit guard**: `isLoading` was only used visually (loading spinner) but the button remained clickable and `create()`/`edit()` didn't check `isLoading.value` at entry. This could cause concurrent async calls that interfere with each other's navigation.

3. **`Get.back()` was unreliable for navigation in `create()`**: The `edit()` method was already using `Get.until((route) => Get.currentRoute == '/menu/product')` (from Fix 1), but `create()` still used `Get.back()` which is fragile — if a snackbar was still visible (treated as a route), `Get.back()` could dismiss the wrong route.

**Changes made**:
- **`product_add_controller.dart` - `create()` method**:
  - Added `if (isLoading.value) return;` guard at the start to prevent double-submit
  - Changed `Get.back()` to `Get.until((route) => Get.currentRoute == '/menu/product')` for reliable navigation
  - Added `else` branch in catch block to show generic error toast (`'global_error_occurred'.tr`) for non-ValidationException errors, with `debugPrint` for debugging
- **`product_add_controller.dart` - `edit()` method**:
  - Added `if (isLoading.value) return;` guard at the start
  - Replaced `debugPrint(e.toString())` with proper `else` branch showing generic error toast for non-ValidationException errors
- **`filled_button.dart` - `MyFilledButton`**:
  - Changed `onPressed` from `VoidCallback` (required, non-null) to `VoidCallback?` (optional, nullable) to allow disabling the button when `null` is passed
- **`product_form.dart` - `ProductForm`**:
  - Updated save button to disable (`onPressed: null`) when `isLoading.value` is true, preventing UI-level double-clicks
- **`messages.dart`**:
  - Added `'global_error_occurred'` translation key for both English and Indonesian
